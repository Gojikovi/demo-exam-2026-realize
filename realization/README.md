## 2) Proxmox VE в виде виртуальной машины

Proxmox устанавливается как виртуальная машина (например, через VirtualBox или VMware).

Плюсы:

* всё находится на одном компьютере
* ближе к реальному экзамену

Минусы:

* сложнее в реализации
* требует больше ресурсов

Схема работы: 

```
Хост система (реальное железо, ПК/ноутбук)
            ↓ 
Гипервизор 2-го типа (hosted) — VMware Workstation
            ↓
Виртуальная машина с Proxmox VE (гипервизор 1-го типа внутри ВМ)
            ↓
Виртуальные машины (KVM)
```

В данной конфигурации используется вложенная виртуализация (nested virtualization). При вложенной виртуализации гипервизор хоста (например, Hyper-V, KVM, VMware) эмулирует/пробрасывает поддержку виртуализации гостевой системе. Это позволяет гостевой виртуальной машине функционировать как гипервизору и создавать собственные виртуальные машины.

VMware Workstation выступает в роли хост-гипервизора второго типа (условно "виртуальный компьютер") и предоставляет виртуальное оборудование для запуска Proxmox VE (условно "сервер виртуализации внутри этого компьютера").

Алгоритм настройки:
- Устанавливаем VMware Workstation на хостовую систему
- Создаём виртуальную машину — в неё будет установлен Proxmox VE. В настройках ВМ в VMware Workstation обязательно ставим галочку на "Virtualize Intel VT-x/EPT or AMD-V/RVI". Без этой галочки виртуальные машины из Proxmox VE просто не запустяться - KVM не сможет использовать аппаратную виртуализацию.
- Устанавливаем Proxmox VE
- После этого в виртуальной машине с Proxmox VE разворачивается стенд демоэкзамена (6 виртуальных машин)

## Перепробованы варианты решений с изменением конфигурации ВМ, но тщетно

### Заключение по проблеме запуска EcoRouter

### Исходные условия

* Гипервизор: VMware Workstation Pro
* Внутри: Proxmox VE (nested virtualization)
* Проблемная ВМ: EcoRouter
* CPU: Intel Core i5-11400H

### Симптом

* EcoRouter зависает на этапе:

kexec_core: Starting new kernel
* Дальнейшая загрузка отсутствует

### Проведённая диагностика

Проверено и исключено:

* Nested virtualization включён и работает
* CPU type: host
* Hyper-V отключён
* Конфигурации ВМ идентичны рабочим стендам
* Проверены CPU flags (включение/исключение)
* Использованы hidden=1, q35
* Исключён invtsc
* Проверены args и чистый конфиг

Дополнительно:

* На других CPU (Intel Core i5-12400F, AMD Ryzen 3 3200G) стенд работает
* В том же VMware Workstation Pro без Proxmox — EcoRouter запускается

### Причина

Ограничение вложенной виртуализации:

VMware Workstation Pro → Proxmox VE → EcoRouter

В данной конфигурации:

* VMware не пробрасывает необходимые MSR/VMX-функции CPU
* EcoRouter использует kexec (перезапуск ядра)
* второе ядро не может корректно инициализировать CPU

В результате возникает зависание на этапе kexec

## Важное уточнение

Проблема:

* не связана с конфигурацией ВМ
* не связана с Proxmox
* не связана с самим EcoRouter

Это ограничение конкретной связки:

* nested virtualization
* мобильный CPU Intel Core i5-11400H
* реализация виртуализации в VMware

### Вывод

Запуск EcoRouter в данной архитектуре невозможен (или крайне нестабилен)

## Имейте это ввиду, если проворачиваете на ноутбуке

## В итоге, использовал ПК на Ryzen 3 3200G. Поставил Proxmox VE в VMware Workstation. А с ноутбука подключился к веб-интерфейсу. Расскажу как настроить интерфейсы в VMware и виртуальной машине Proxmox для такого варианта работы

### 1. Настройка сетевых адаптеров в VMware

У виртуальной машины Proxmox должно быть 2 сетевых адаптера:

Адаптер 1 (Интернет)

Тип: NAT (VMnet8)

Назначение: доступ в интернет внутри Proxmox и ВМ

Адаптер 2 (Доступ с других устройств)

Тип: Bridged

Назначение: доступ к Proxmox из локальной сети (с ноутбука)

Важно: Bridged-адаптер должен быть привязан к физической сети (Wi-Fi или Ethernet), в которой находится устройство, с которого ты будешь подключаться.

### 2. Схема сети

После настройки получается:

ПК (хост): 192.168.1.x

Ноутбук: 192.168.1.x

Proxmox (через Bridged): 192.168.1.x

Все устройства в одной подсети, поэтому видят друг друга.

### 3. Настройка сети в Proxmox

Внутри Proxmox нужно настроить bridge-интерфейс. Запускаем виртуалку и открываем:

```
nano /etc/network/interfaces

# Пример конфигурации
auto lo
iface lo inet loopback

iface ens32 inet manual
iface ens33 inet manual

auto vmbr0
iface vmbr0 inet dhcp
    bridge-ports ens33
    bridge-stp off
    bridge-fd 0
```

### 4. Применение настроек

```
systemctl restart networking.service
или
reboot
```

### 5. Подключение к веб-интерфейсу

```
ip a
# Смотрим какой ip-адрес получил ens33 (Bridged)
С подключаемого устройства вводим в поисковую строку браузера:
# Пример
https://192.168.1.200:8006
```

## 3) Proxmox VE на реальном железе

Установка на отдельный компьютер.

Схема работы:

* один компьютер — сервер с Proxmox и виртуальными машинами
* второй компьютер — для подключения и настройки через веб-интерфейс

Это максимально приближено к реальному демоэкзамену.

### Важные моменты по Proxmox

Для установки потребуется отдельный диск.

Если у вас на компьютере уже стоит Windows и вы не хотите её сносить, то:

* Windows не видит файловую систему Linux (ext4)
* Linux не работает с NTFS как с основной системой

В итоге:

* диск полностью форматируется под ext4
* получается dual boot

Поэтому для меня этот 3 вариант не подходит и использовать его я точно не буду.

Также:

для нормальной работы Proxmox желательно проводное подключение.

По информации из сети, через Wi-Fi возможны проблемы с виртуальными коммутаторами и стабильностью стенда.

## Почему VMware, а не VirtualBox

1. По моим наблюдениям, виртуальные машины работают быстрее
2. С 11 ноября 2024 года VMware Workstation стал бесплатным
3. У меня есть образ EcoRouterOS в формате VMDK

VirtualBox такой формат напрямую не использует, поэтому выбор очевиден.

### Шаг 21 (НЕОБЯЗАТЕЛЬНО)

Данный шаг является необязательным, но упростит работу с HQ-RTR и BR-RTR как и в среде VMware, так и в Proxmox. Особенно когда работаете со всеми запущенными ВМ.

> [!TIP]
> По умолчанию, спустя 10 минут бездействия, EcoRouter разлогинивает вас с учётной записи, что заставляет вас заново вводить логин и пароль
>
> Это поведение можно изменить, прописав данные настройки в линиях консолей console 0 и vty 0 39
> 
> или vty 0 871 (сразу всем 872 виртуальным линиям)
> 
> Просмотреть линии vty можно через show run

```
conf t
line con 0
exec-timeout 0 0
exit
line vty 0 871
exec-timeout 0 0
exit
write memory
```

![Шаг 21](../pictures-r/step21-routers.png)

После пишем exit, перезаходим для применения тайм-аутов

write memory - команда для записи изменений в маршрутизатор. 

Для большинства команд используются сокращённые версии (en, conf t, show run, wr mem, ip a и т.д.)
